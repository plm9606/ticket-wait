# Refine Plan

플랜 모드에서 생성된 계획을 재검토하여 해상도를 높입니다. 명확하지 않은 부분을 찾아 사용자에게 질문하고, 대용량 트래픽 가용성/성능/엣지케이스를 점검하여 실행 가능한 수준의 구체적인 플랜으로 만듭니다.

## 인수가 없을 때 안내

인수가 비어 있으면 `.claude/plans/` 디렉토리에서 가장 최근 plan 파일을 자동으로 찾습니다. 파일이 없으면 다음을 안내합니다:

---

**플랜 파일을 찾을 수 없습니다.** 아래 방법 중 하나로 진행해 주세요:

| 방법 | 예시 |
|------|------|
| **파일 경로 지정** | `/refine-plan .claude/plans/my-plan.md` |
| **플랜 먼저 생성** | plan 모드에서 플랜을 먼저 작성한 뒤 이 스킬을 실행하세요 |

---

## 실행 순서

### 1단계: 플랜 파일 로드

인수로 경로가 주어지면 해당 파일을 읽습니다. 없으면 자동 탐색합니다:

```text
.claude/plans/*.md 중 가장 최근 수정된 파일
```

파일을 찾으면 전체 내용을 읽어들입니다.

### 2단계: 플랜 구조 분석

플랜의 전체 구조를 파악합니다:
- 몇 개의 단계로 구성되어 있는지
- 각 단계가 어떤 작업을 하는지
- 단계 간 의존 관계가 있는지
- 변경 대상 파일/모듈 목록

### 3단계: 코드베이스 크로스체크

플랜에서 언급된 파일, 클래스, 메서드가 실제로 존재하는지 확인합니다.

- 플랜이 참조하는 파일 경로가 실제 존재하는지 Glob/Grep으로 검증
- 수정 대상 클래스/메서드의 현재 시그니처를 읽어서 플랜의 가정과 맞는지 확인
- 의존하는 인터페이스나 타입이 실제 코드와 일치하는지 검증

불일치가 발견되면 리파인 결과에 포함합니다.

### 4단계: 명확성 검토

플랜의 각 단계를 아래 기준으로 검토하여, 모호하거나 결정이 필요한 부분을 수집합니다:

**모호성 체크리스트:**
- "적절히", "필요에 따라", "등"처럼 구체적 기준 없는 표현이 있는가
- 두 가지 이상 해석이 가능한 문장이 있는가
- 구현 방식이 여러 가지인데 하나로 특정되지 않은 부분이 있는가
- 새로운 인터페이스/타입 정의가 필요한데 스펙이 빠져 있는가
- 에러/실패 시 동작이 정의되지 않은 단계가 있는가
- 변경 범위가 불명확한 단계가 있는가 (어떤 파일, 어떤 메서드를 수정하는지)

**의사결정 필요 항목:**
- 비즈니스 로직 선택지가 있는데 어느 쪽인지 명시되지 않은 경우
- 기존 코드와 새 코드 간 마이그레이션 전략이 빠진 경우
- 외부 시스템 연동 시 타임아웃, 재시도, 폴백 정책이 미정인 경우
- 데이터 모델 변경 시 하위 호환성 처리 방안이 빠진 경우

### 5단계: 가용성 및 성능 검토

대용량 트래픽 환경에서의 안정성을 기준으로 플랜을 검토합니다:

**데이터베이스:**
- 새로운 쿼리에 적절한 인덱스가 존재하는가 (또는 인덱스 추가 플랜이 있는가)
- N+1 쿼리 발생 가능성이 있는가
- 대량 데이터 조회 시 페이징/커서 전략이 있는가
- Writer/Reader DB 라우팅이 올바른가 (FromRead 규칙 준수 여부)
- 트랜잭션 범위가 필요 이상으로 넓지 않은가

**외부 시스템 연동:**
- gRPC/REST 호출에 deadline/timeout이 설정되어 있는가
- 외부 시스템 장애 시 폴백/서킷브레이커 전략이 있는가
- 동기 호출이 비동기로 전환 가능한 부분은 없는가

**동시성 및 캐싱:**
- 동시 요청 시 race condition 가능성이 있는가
- 캐시 도입이 필요한 hot path가 있는가
- 캐시 도입 시 invalidation 전략이 있는가

**배포 안전성:**
- 롤백 가능한 방식인가 (DB 스키마 변경이 하위 호환되는가)
- feature flag 없이 배포해도 안전한가
- 배포 순서에 의존성이 있는가 (예: IDL 먼저 배포 필요)

### 6단계: 사용자 질문

4단계, 5단계에서 수집된 항목을 정리하여 사용자에게 질문합니다. AskUserQuestion을 사용하여 구조화된 질문을 합니다.

질문 우선순위:
1. **Blocker**: 이 결정 없이는 구현을 시작할 수 없는 항목
2. **Important**: 구현 중 막힐 가능성이 높은 항목
3. **Nice-to-have**: 있으면 좋지만 합리적 기본값으로 진행 가능한 항목

Nice-to-have 항목은 질문 없이 제안 형태로 포함합니다. Blocker와 Important만 질문합니다.

질문이 없을 경우(플랜이 이미 충분히 구체적이면) 이 단계를 건너뛰고 바로 결과 보고로 넘어갑니다.

### 7단계: 리파인된 플랜 작성

사용자의 답변을 반영하여 원본 플랜 파일을 업데이트합니다:

- 모호했던 부분을 구체적인 구현 지시로 교체
- 누락된 에러 처리, 성능 고려사항을 단계에 추가
- 코드베이스 크로스체크에서 발견된 불일치를 수정
- 각 단계에 변경 대상 파일과 예상 변경 내용을 명시

### 8단계: 리파인 결과 보고

아래 형식으로 결과를 보고합니다:

```text
## Plan Refinement 결과

### 변경 요약
- 명확화: N개 항목
- 가용성/성능 보완: N개 항목
- 코드베이스 불일치 수정: N개 항목

### 명확화된 항목
| # | 원본 | 리파인 후 | 근거 |
|---|------|----------|------|
| 1 | ... | ... | ... |

### 가용성/성능 보완 항목
| # | 항목 | 리스크 | 추가된 대응 |
|---|------|--------|-----------|
| 1 | ... | ... | ... |

### 남은 리스크 (수용 가능)
- ...
```

## 주의사항

- 플랜 파일 원본은 반드시 백업 없이 직접 수정합니다 (git으로 복원 가능).
- 사용자에게 질문할 때 한 번에 5개 이하로 제한합니다. 질문이 많으면 우선순위가 높은 것부터 나눠서 질문합니다.
- 코드베이스를 읽을 때 전체를 탐색하지 않고, 플랜에서 언급된 파일/모듈만 확인합니다.
- 프로젝트의 CLAUDE.md에 정의된 규칙(DB 라우팅, 테스트 규칙 등)을 가용성 검토 기준에 포함합니다.
